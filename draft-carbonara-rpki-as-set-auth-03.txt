Abstract

   This document specifies the RPKI AS-SET Authorization (RASA)
   protocol, which provides cryptographic verification of AS-SET
   membership using the Resource Public Key Infrastructure (RPKI).  RASA
   prevents unauthorized AS-SET inclusion, a common vector for massive
   route leaks, while maintaining compatibility with existing Internet
   Routing Registry (IRR) infrastructure.

   RASA delivers immediate operational benefits: network operators gain
   automatic route leak prevention as tools adopt RASA verification,
   integration with PeeringDB enables automated monitoring and offender
   notification, and the combination of RPKI origin validation with AS-
   SET-based AS_PATH filtering reduces reliance on large prefix-lists.
   The protocol is designed for incremental deployment, providing
   security improvements even with partial adoption.

1.  Introduction

   Internet Routing Registry (IRR) databases have long been used to
   generate BGP route filters.  The Routing Policy Specification
   Language (RPSL) [RFC2622] defines the as-set object, which represents
   a set of autonomous systems.  However, IRR-based AS-SETs suffer from
   a fundamental security limitation: membership is controlled solely by
   the AS-SET owner, without cryptographic verification or member
   authorization.




Abrahamsson              Expires 28 August 2026                 [Page 3]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   This creates two significant problems:

   1.  *Unauthorized Inclusion:* An AS-SET owner can include any
       autonomous system without authorization, potentially allowing
       route leaks or policy violations.

   2.  *Transitive Chain Attacks:* AS-SET expansion is transitive - if
       AS-SET A contains AS-SET B, and B contains AS C, then A
       implicitly contains C.  This creates a chain of trust where
       compromising any link compromises the entire chain.

   RPKI AS-SET Authorization (RASA) addresses these limitations by:

   *  Providing cryptographic signatures for AS-SET membership via RPKI

   *  Enabling verification during AS-SET expansion

   *  Supporting "do-not-inherit" semantics to prevent transitive
      attacks

   *  Maintaining backward compatibility with existing IRR
      infrastructure

2.  Operational Benefits

   RASA provides significant operational advantages for network
   operators deploying the protocol.  These benefits accrue
   automatically as tools adopt RASA verification, without requiring
   flag-day deployments or wholesale infrastructure changes.

2.1.  Automatic Security Improvements

   Unlike routing security mechanisms that require coordinated
   deployment, RASA provides incremental security benefits as individual
   tools and services upgrade to support RASA verification:

   *  Route filter generators that support RASA automatically reject
      unauthorized AS-SET inclusions, preventing route leaks without
      operator intervention.

   *  Operators gain cryptographic assurance of AS-SET membership as
      more autonomous systems publish RASA objects, even if adoption is
      partial.

   *  Existing IRR-based workflows continue to function during the
      transition, with security improvements appearing gradually as RASA
      adoption increases.




Abrahamsson              Expires 28 August 2026                 [Page 4]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   This automatic improvement model means that upgrading a single
   component (e.g., a route server or filter generation tool) to support
   RASA verification immediately improves security for all AS-SETs
   protected by RASA, regardless of whether other operators have
   upgraded.

2.2.  PeeringDB Integration

   RASA enables third-party monitoring services to traverse publicly
   available AS-SET data and identify discrepancies between claimed and
   cryptographically authorized memberships.  This capability
   facilitates proactive notification to network operators when
   unauthorized AS numbers appear in their AS-SETs.

   For example, a monitoring service could:

   1.  Query PeeringDB for AS-SET relationships and member lists.

   2.  Retrieve published RASA objects from the RPKI repository.

   3.  Identify ASNs listed in an AS-SET that have not authorized
       inclusion via RASA-AUTH objects.

   4.  Automatically notify both the AS-SET owner and the affected
       autonomous system of the discrepancy.

   This automated monitoring and notification capability reduces the
   time between unauthorized inclusion and remediation, limiting the
   window for potential route leaks.  It also provides an audit trail
   that can be used to identify persistent offenders or systemic issues
   in AS-SET management practices.

2.3.  Reduced Configuration Complexity

   RASA, combined with existing RPKI Route Origin Validation (RPKI-ROV),
   enables a significant reduction in prefix-list filter complexity.
   Traditional route filtering requires maintaining large prefix-lists
   containing all customer prefixes.  With RASA-verified AS-SETs and
   RPKI origin validation, operators can instead use AS_PATH filters
   derived from cryptographically verified AS-SET memberships.

   The operational advantages of this approach include:

   *  Reduced router configuration state: AS_PATH filters are typically
      orders of magnitude smaller than equivalent prefix-lists.






Abrahamsson              Expires 28 August 2026                 [Page 5]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   *  Decreased filter update frequency: Rather than updating filters
      every time a customer announces a new prefix, operators need only
      update when AS-SET membership changes.

   *  Lower risk of stale entries: RPKI provides cryptographic
      expiration and revocation, eliminating stale prefix-list entries.

   *  Improved convergence times: Shorter filter lists can be processed
      more quickly during BGP route processing.

   For example, rather than maintaining a prefix-list containing
   hundreds or thousands of customer prefixes, an operator may configure
   a single AS_PATH filter permitting only authorized AS numbers to
   appear in the path, relying on RPKI to validate the origin AS.

2.4.  Deployment Advantages

   RASA is designed to minimize deployment friction:

   *  No flag day: Incremental adoption is possible and beneficial even
      with partial deployment.

   *  IRR compatibility: Existing RPSL objects and IRR infrastructure
      remain functional during and after RASA deployment.

   *  No IRR changes: The protocol does not require changes to IRR
      database schemas or protocols.

   *  Tool upgradability: Existing filter generation tools can be
      updated to support RASA without architectural changes.

   *  ASPA complementarity: RASA works alongside ASPA for comprehensive
      routing security.

4.  Design Rationale

   The RASA design is informed by several existing mechanisms and their
   limitations:

4.1.  IRR-Based Approaches

   IRR-based AS-SETs rely on the AS-SET owner's mnt-by attribute for
   authentication.  This has several weaknesses:

   *  No cryptographic verification of membership

   *  Members cannot prevent unauthorized inclusion

   *  Transitive inclusion cannot be controlled

   *  Authentication relies on passwords or PGP keys in IRR

   The member-of-as-set proposal [I-D.member-of-as-set] addresses some
   of these issues but requires the IRR database to support a new object
   type and does not provide cryptographic verification.

   Prior work by Snijders and Korsb√§ck proposed ASGroup [I-D.asgroup],
   an RPKI-based AS grouping mechanism with a complex opt-out mechanism.
   RASA takes a different approach with a simplified model that
   prioritizes IRR compatibility and incremental deployment.  The opt-
   out mechanism in ASGroup proved difficult to reason about
   operationally and added significant complexity to both specification
   and implementation.

4.2.  ASPA Comparison

   Autonomous System Provider Authorization (ASPA) [I-D.aspa] provides
   cryptographic verification of provider relationships.  ASPA proves
   that a Customer AS (CAS) has authorized a Provider AS (PAS) as its
   upstream.

   However, ASPA has a different scope than RASA:

   *  ASPA proves provider-customer relationships

   *  RASA proves AS-SET membership

   *  ASPA is used during BGP path validation

   *  RASA is used during route filter generation





Abrahamsson              Expires 28 August 2026                 [Page 7]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   While both use RPKI, they serve complementary purposes and are not
   interchangeable.

4.3.  Design Goals

   RASA is designed with the following goals:

   1.  *Cryptographic Verification:* Use RPKI signing to ensure AS-SET
       membership cannot be spoofed

   2.  *Member Protection:* Prevent unauthorized inclusion of ASNs in
       AS-SETs

   3.  *Chain Control:* Allow AS-SET owners to prevent transitive
       inclusion attacks

   4.  *Backward Compatibility:* Work alongside existing IRR
       infrastructure

   5.  *Incremental Deployment:* Provide value even with partial
       adoption

5.  Delegation Model

   RASA supports flexible delegation models allowing AS-SET owners to
   manage publishing responsibilities.  This section describes how AS-
   SET owners can authorize other entities to publish RASA objects on
   their behalf.

5.1.  Delegation Overview

   In the RASA delegation model, the AS-SET owner (represented by their
   RPKI certificate) may either publish RASA objects directly or
   delegate publishing authority to other entities.  This flexibility
   accommodates various operational models, from direct management by
   the AS-SET owner to delegated management by customers or third-party
   services.

   The delegation model draws inspiration from established patterns in
   RPKI delegation (RFC 8183) and IRR maintainer structures (RFC 2725),
   adapted for the specific requirements of AS-SET authorization.

5.2.  Direct Publishing

   In the direct publishing model, the AS-SET owner creates and
   publishes RASA-SET objects using their own RPKI certificate.  This is
   the simplest model and is appropriate when:




Abrahamsson              Expires 28 August 2026                 [Page 8]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   *  The AS-SET owner manages all member relationships directly

   *  The AS-SET is small and infrequently changed

   *  The AS-SET owner has operational control of RPKI publication

   For example, a Tier-1 provider publishing their customer AS-SET would
   typically use direct publishing, as they maintain direct
   relationships with all customer ASes.

5.3.  Delegated Publishing

   In the delegated publishing model, the AS-SET owner issues an
   authorization token that grants a delegated entity (e.g., a customer
   AS) the right to publish RASA-AUTH objects on behalf of the AS-SET
   owner.  This model is appropriate when:

   *  Customers manage their own AS-SET membership authorization

   *  The AS-SET owner wishes to offload authorization management

   *  Complex multi-party relationships require distributed control

   The delegation mechanism uses RPKI-signed authorization tokens
   similar to the model defined in RFC 8183.  An AS-SET owner creates a
   delegation token that specifies:

   Delegated Entity:  The ASN or entity receiving delegation authority

   Scope:  The specific AS-SET(s) covered by the delegation

   Constraints:  Any limitations on the delegated authority (e.g.,
      validity period)

   Delegated entities MAY publish RASA-AUTH objects for the authorized
   AS-SET using their own RPKI certificate, but these objects MUST
   reference the delegation token issued by the AS-SET owner.

5.4.  Delegation Validation

   When validating a delegated RASA object, validators MUST verify:

   1.  The RASA object is cryptographically valid (signed by a
       certificate in the RPKI chain).

   2.  If the signing certificate's ASID matches the containingAS, the
       object is directly published and requires no delegation
       validation.



Abrahamsson              Expires 28 August 2026                 [Page 9]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   3.  If the signing certificate's ASID does not match the
       containingAS, the validator MUST locate a valid delegation token
       issued by the containingAS that authorizes the signing entity.

   4.  The delegation token MUST be valid (not expired or revoked) and
       MUST cover the AS-SET in question.

   This validation chain ensures that even with delegation, the AS-SET
   owner maintains ultimate control through the ability to revoke
   delegation tokens or let them expire.

5.5.  Revocation

   Delegated authority can be revoked through several mechanisms:

   *  Expiration: Delegation tokens include a notAfter timestamp; after
      this time, the delegation is automatically invalid.

   *  RPKI Revocation: If the delegated entity's RPKI certificate is
      revoked (e.g., due to compromise or resource transfer), all RASA
      objects signed by that certificate become invalid.

   *  Delegation Withdrawal: The AS-SET owner MAY publish an updated
      delegation token with a shorter validity period, effectively
      forcing re-authorization.

5.6.  Delegation Examples

   Example 1: Customer-Managed Authorization

   AS-SET Owner (AS2914) delegates to customer AS64496:
     Delegation Token (signed by AS2914):
       delegatedTo: AS64496
       scope: AS2914:AS-GLOBAL
       notBefore: 2024-01-01T00:00:00Z
       notAfter: 2025-01-01T00:00:00Z

   Customer AS64496 publishes RASA-AUTH:
     authorizedAS: 64496
     authorizedIn: [AS2914:AS-GLOBAL]
     signedBy: AS64496 certificate
     references: [delegation-token-from-AS2914]

   Example 2: Third-Party AS-SET Management Service

   Multiple AS-SET owners delegate to management service AS-SET-MGMT:
     AS6939 delegates: AS6939:AS-HURRICANE
     AS1299 delegates: AS1299:AS-TWELVE99

6.  RASA Objects

   RASA defines two separate RPKI object types that work together to
   provide complete AS-SET authorization:

   1.  *RASA-SET:* Defines AS-SET membership (published by AS-SET
       owner).  Replaces the IRR as-set object.

   2.  *RASA-AUTH:* Defines which AS-SETs may include an ASN or AS-SET
       (published by ASN or AS-SET owner).  Provides cryptographic
       authorization for AS-SET inclusion.

   These objects are intentionally separate to maintain clear separation
   of concerns and allow independent deployment.  The two-object design
   is discussed further in Section 4.3.

6.1.  RASA-SET Object

   The RASA-SET object defines the membership of an AS-SET.  It is
   published by the AS that owns the AS-SET (the containingAS).

6.1.1.  Object Format

   The RASA-SET object is encoded using CMS [RFC5652] with the
   SignedData content type.  The eContentType is:

   id-rpki-rasa-set OBJECT IDENTIFIER ::= { id-rpki 42 1 }

   The RASA-SET content has the following ASN.1 structure:

   Note: ASID is defined as INTEGER (0..4294967295) per RFC 3779.  The
   OID arc id-rpki will be assigned by IANA during the publication
   process.










Abrahamsson              Expires 28 August 2026                [Page 11]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   RasaFlags ::= BIT STRING {
       doNotInherit(0),
       authoritative(1)
   }

   RasaAuthFlags ::= BIT STRING {
       strictMode(0)
   }

   -- Propagation scope for peer locking
   PropagationScope ::= ENUMERATED {
       unrestricted(0),    -- Can be nested anywhere (default)
       directOnly(1)       -- Only direct inclusion, no nesting/transit
   }

   -- Authorized entry with propagation constraint
   AuthorizedEntry ::= SEQUENCE {
       asSetName        UTF8String,
       propagation      PropagationScope OPTIONAL
       -- If propagation is absent, defaults to unrestricted
   }

   RasaSetContent ::= SEQUENCE {
     version          [0] INTEGER DEFAULT 0,
     asSetName            UTF8String,
     containingAS         ASID,
     members              SEQUENCE OF ASID,
     nestedSets           SEQUENCE OF UTF8String OPTIONAL,
     irrSource            UTF8String OPTIONAL,
     flags                RasaFlags OPTIONAL,
     notBefore            GeneralizedTime,
     notAfter             GeneralizedTime
   }

6.1.2.  Field Definitions

   version:  The version of the RASA-SET format.  This document defines
      version 0.

   asSetName:  The name of the AS-SET (e.g., "AS1299:AS-TWELVE99").
      MUST start with "AS-" per RFC2622.

   containingAS:  The ASN of the AS that owns this AS-SET.  Must match
      the AS in hierarchical names (e.g., 1299 for AS1299:AS-TWELVE99).

   members:  List of ASNs that are members of this AS-SET.  Each ASID
      must be unique and sorted.  The containingAS MAY be included.




Abrahamsson              Expires 28 August 2026                [Page 12]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   nestedSets:  OPTIONAL list of other AS-SET names that are members of
      this AS-SET.  Enables recursive AS-SET expansion.  Each nested set
      will be resolved by looking up its RASA-SET (or falling back to
      IRR if no RASA-SET exists).

   irrSource:  OPTIONAL authoritative IRR database for this AS-SET
      (e.g., "RADB", "RIPE").  Prevents namespace ambiguity.

   flags:  Optional flags: doNotInherit (prevent transitive inclusion),
      authoritative (RASA-SET replaces IRR data).

   notBefore, notAfter:  Validity period per RPKI policies.

6.2.  RASA-AUTH Object

   The RASA-AUTH object contains an authorization statement from an
   autonomous system or AS-SET owner, granting permission for inclusion
   in specified parent AS-SETs.  It is published by the ASN or AS-SET
   owner to control inclusion authorization.

6.2.1.  Object Format

   The RASA-AUTH object is encoded using CMS [RFC5652] with the
   SignedData content type.  The eContentType is:

   id-rpki-rasa-auth OBJECT IDENTIFIER ::= { id-rpki 42 2 }

   The RASA-AUTH content has the following ASN.1 structure:

   RasaAuthContent ::= SEQUENCE {
     version          [0] INTEGER DEFAULT 0,
     authorizedEntity CHOICE {
        authorizedAS        ASID,
        authorizedSet       UTF8String
     },
     authorizedIn         SEQUENCE OF AuthorizedEntry,
     flags                RasaAuthFlags OPTIONAL,
     notBefore            GeneralizedTime,
     notAfter             GeneralizedTime
   }

   -- Note: authorizedIn always uses AuthorizedEntry. The propagation field
   -- is OPTIONAL - if absent, defaults to unrestricted.

6.2.2.  Field Definitions

   version:  The version of the RASA-AUTH format.  This document defines
      version 0.  Future versions may add new features.



Abrahamsson              Expires 28 August 2026                [Page 13]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   authorizedEntity:  CHOICE between authorizedAS and authorizedSet:

      *  *authorizedAS:* The ASN that is authorizing inclusion.  Use for
         individual ASNs.

      *  *authorizedSet:* The AS-SET name that is authorizing inclusion.
         Use for AS-SETs that may be included in other AS-SETs.

   authorizedIn:  Each entry is an AuthorizedEntry containing asSetName
      (the AS-SET name) and optionally propagation (PropagationScope).
      If propagation is absent, it defaults to unrestricted.  The
      propagation field signals BGP import policy preferences:
      unrestricted (default) has no special semantics; directOnly
      advises the containing AS to only accept routes with this ASN from
      direct BGP sessions (peer lock signal).  An empty list means the
      ASN refuses inclusion in any AS-SET.

   flags:  Optional flags controlling authorization behavior.
      strictMode: If set, AS-SETs NOT in this list MUST NOT include this
      ASN (reject rather than warn).

   notBefore, notAfter:  Validity period per RPKI policies.

6.3.  Design Overview

   RASA defines two separate RPKI object types:

   RASA-SET:  Published by the AS-SET owner (e.g., AS1299).  Contains
      the AS-SET membership list (ASNs and nested AS-SETs).

   RASA-AUTH:  Published by the ASN or AS-SET owner.  Authorizes
      inclusion of the ASN (via authorizedAS) or AS-SET (via
      authorizedSet) in specific parent AS-SETs.

   This separation maintains cryptographic boundaries: only the ASN
   owner can authorize inclusion, preventing AS-SET owners from claiming
   unauthorized membership.

7.  RASA Verification

   RASA objects are verified using standard RPKI validation procedures.

7.1.  Signature Validation

   The RASA object MUST be validated according to [RFC6488]:

   1.  Verify the CMS signature and certificate chain per [RFC6488].




Abrahamsson              Expires 28 August 2026                [Page 14]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   2.  Verify the certificate's subject matches the containingAS (for
       RASA-SET) or authorizedAS/authorizedSet (for RASA-AUTH).

7.2.  Content Validation

   In addition to signature validation, the RASA content MUST be
   validated for conformance to this specification:

   1.  The asSetName MUST start with "AS-" (case-insensitive).  RASA
       objects with asSetName not conforming to RFC2622 naming
       conventions MUST be rejected.  This prevents namespace collisions
       between AS-SETs and autonomous system numbers.

   2.  For hierarchical AS-SET names (e.g., "AS1299:AS-TWELVE99"), the
       ASN component MUST match the containingAS field.

   3.  The members list MUST contain only valid ASNs (no AS-SET names in
       the members list; nested AS-SETs are handled via expansion).

   4.  All ASIDs in the members list MUST be sorted in ascending order
       with no duplicates.

7.3.  AS-SET Expansion with RASA

   When expanding an AS-SET for route filter generation, RPKI-aware
   tools SHOULD perform the following algorithm:

   1.  Query RPKI for a RASA object for the AS-SET.  If a valid RASA
       exists and the authoritative flag is set, use ONLY the RASA
       member list.  Do NOT query IRR for this AS-SET.  The IRR object
       is treated as deprecated/non-authoritative.

   2.  If a valid RASA exists but authoritative is NOT set, determine
       the IRR database(s) to query: if irrSource is specified, query
       ONLY that database.  If irrSource is absent, query all configured
       IRR databases.  Merge the RASA member list with the IRR member
       list.

   3.  If no valid RASA exists, query IRR using the same database
       selection logic (irrSource if present, otherwise all databases).











Abrahamsson              Expires 28 August 2026                [Page 15]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   4.  For each nested AS-SET in the member list, first verify if the
       nested AS-SET has published a RASA-AUTH (with authorizedSet).  If
       so, verify the current AS-SET name is in the authorizedIn list.
       If not authorized, skip the nested AS-SET and log a warning.  If
       authorized (or no RASA-AUTH exists), recurse using the same
       algorithm.  If the parent AS-SET has doNotInherit set, include
       only the AS-SET reference (not its members) in the expansion.
       Note: The propagation field in RASA-AUTH is for BGP import policy
       signaling and does NOT affect AS-SET expansion.

   5.  For each member ASN, verify if a RASA object exists that lists
       this AS-SET as authorized.  If the ASN has published a RASA and
       this AS-SET is not in the authorized list, exclude the ASN.  The
       strictMode flag controls logging: if TRUE, log a security event;
       if FALSE, log a warning.

   The irrSource field prevents namespace ambiguity attacks where the
   same AS-SET name exists in multiple IRR databases with different
   members.  When specified, tools MUST NOT query other IRR databases.

8.  Conflict Resolution

   When both RASA-SET and RASA-AUTH objects exist for the same AS-SET
   membership claim, validators MUST apply the following resolution
   rules to determine the authoritative membership list.

8.1.  Priority Rules

   RASA-AUTH (member's intent) takes precedence over RASA-SET (owner's
   claim).  The rationale is that an AS should have control over which
   AS-SETs include it, as inclusion affects their routing security.

   *Resolution Algorithm:*

   1.  For each member ASN claimed in RASA-SET.members, check if a RASA-
       AUTH object exists signed by that ASN.

   2.  If RASA-AUTH exists: verify this AS-SET name is in the RASA-
       AUTH.authorizedIn list.  If present, include the ASN.  If the AS-
       SET is not in authorizedIn, exclude the ASN.  Note: The
       propagation field (directOnly/unrestricted) does NOT affect AS-
       SET expansion decisions.  It is an advisory signal for BGP import
       policy at the containing AS's discretion.

   3.  If RASA-AUTH exists and strictMode is set to TRUE, exclude the
       ASN and log a security event (unauthorized inclusion).





Abrahamsson              Expires 28 August 2026                [Page 16]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   4.  If RASA-AUTH exists and strictMode is FALSE, exclude the ASN but
       log a warning (member may not want inclusion).

   5.  If no RASA-AUTH exists, include the ASN if the AS-SET owner's
       RASA-SET is valid.

9.  Circular Reference Detection

   RASA-SET objects support nested AS-SET references via the nestedSets
   field.  This creates the potential for circular references if an AS-
   SET includes itself, directly or indirectly.

   Validators MUST detect and handle circular references to prevent
   infinite loops during AS-SET expansion.

9.1.  Detection Algorithm

   During AS-SET expansion, validators MUST maintain a "seen" set
   containing all AS-SET names encountered in the current expansion
   chain.

   1.  Initialize an empty set 'seen' before expanding an AS-SET.

   2.  Before expanding any AS-SET (via RASA-SET or IRR), check if its
       name is already in 'seen'.

   3.  If yes: circular reference detected.  Stop expanding this branch
       and log a warning.  Do not include any members from the circular
       AS-SET.




Abrahamsson              Expires 28 August 2026                [Page 19]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   4.  If no: add the AS-SET name to 'seen' and continue expansion.

   5.  When backtracking from a nested AS-SET, remove its name from
       'seen' to allow valid reuse in different branches.

9.2.  Circular Reference Example

   RASA-SET (AS2914):
     asSetName: "AS2914:AS-GLOBAL"
     members: [AS64496]
     nestedSets: ["AS64496:AS-SET"]

   RASA-SET (AS64496):
     asSetName: "AS64496:AS-SET"
     members: [AS398465]
     nestedSets: ["AS2914:AS-GLOBAL"]  -- circular!

   Expansion trace:
   1. Start: AS2914:AS-GLOBAL (seen={})
   2. Add AS2914:AS-GLOBAL to seen
   3. Expand AS64496:AS-SET (seen={AS2914:AS-GLOBAL})
   4. Add AS64496:AS-SET to seen
   5. Attempt to expand AS2914:AS-GLOBAL
   6. Detect: AS2914:AS-GLOBAL already in seen
   7. Log warning, stop expansion, return [AS64496] only

11.  Security Considerations

   RASA provides cryptographic assurance of AS-SET membership, but
   several security considerations apply:

11.1.  Partial Deployment

   During partial deployment, some AS-SETs will not have RASA objects.
   Tools SHOULD:



Abrahamsson              Expires 28 August 2026                [Page 21]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   *  Default to IRR data when RASA is unavailable (backward
      compatibility)

   *  Provide configuration options to require RASA for specific AS-SETs
      or all AS-SETs

   *  Log warnings when falling back to IRR for high-value AS-SETs

11.2.  Chain Attack Mitigation

   The doNotInherit flag mitigates transitive chain attacks.  However,
   if an AS-SET owner does not set this flag, their members remain
   vulnerable to transitive inclusion.  AS-SET owners should carefully
   consider whether their AS-SET should be transitively inheritable.

11.3.  Chain-of-Trust Limitations

   RASA provides cryptographic verification of individual AS-SETs, but a
   fundamental limitation remains: AS-SET expansion is transitive.  When
   AS-SET A contains AS-SET B, and B contains AS-SET C, then A
   implicitly contains all members of C.

   This creates a chain-of-trust problem: an attacker who compromises
   any AS-SET in the expansion chain can include arbitrary ASNs.  For
   example, if a malicious actor creates AS-EVIL containing AS15169:AS-
   GOOGLE, and AS15169:AS-GOOGLE has not set the doNotInherit flag, then
   AS-EVIL inherits all Google ASNs.

   *Critical Requirement:* For full security, every AS-SET in the
   expansion chain must protect itself.  If any AS-SET in the chain
   lacks RASA protection or does not set doNotInherit, the entire chain
   remains vulnerable to transitive attacks.

   This limitation is not unique to RASA - it affects all AS-SET-based
   routing, including IRR-based AS-SETs.  The difference is that RASA
   provides a mechanism (doNotInherit) for AS-SET owners to break the
   chain.  Full protection requires widespread adoption at the internet
   core: Tier-1 providers and major content networks must publish RASA
   objects with appropriate flags.

12.  Deployment Considerations

   RASA is designed for incremental deployment.  Content providers and
   Tier-1 operators can publish RASA objects immediately, even before
   widespread tool support.






Abrahamsson              Expires 28 August 2026                [Page 22]

Internet-Draft          RPKI AS-SET Authorization          February 2026


   An alternative approach uses existing ASPA objects to synthesize AS-
   SET membership from provider relationships.  While this requires no
   new object types, it only captures provider-customer relationships
   and lacks authorization semantics.

12.1.  ASPA-RASA Relationship

   ASPA synthesis and RASA serve complementary purposes:

   *  *ASPA-derived AS-SETs* represent provider-customer relationships
      discovered from ASPA objects.  These are dynamic - membership
      changes as ASPAs are published or withdrawn.

   *  *RASA objects* represent explicit AS-SET membership declarations
      with authorization controls.

   Operators may use ASPA-derived AS-SETs for provider-customer
   relationships while using RASA for complex multi-lateral groupings
   requiring authorization.

13.  Operational Guidance

   This section provides practical guidance for network operators
   deploying RASA in production environments.

13.1.  Publishing RASA Objects

   To publish a RASA object, an AS operator must:

   1.  Generate a RPKI certificate for their AS if not already available
       through their Regional Internet Registry (RIR).

   2.  Create the RASA content specifying the AS-SET name, containing
       AS, member list, and appropriate flags.

   3.  Sign the RASA object using their RPKI private key via CMS
       [RFC5652].

   4.  Publish the signed RASA object to their RPKI repository,
       typically via their RIR's publication service or a private
       repository if using delegated RPKI.

   5.  Ensure the RASA object is included in the repository manifest and
       available via rsync or RRDP.







Abrahamsson              Expires 28 August 2026                [Page 23]

Internet-Draft          RPKI AS-SET Authorization          February 2026


13.2.  Consuming RASA Objects

   Operators generating route filters SHOULD integrate RASA validation
   into their existing IRR tools:

   1.  Query RPKI repositories for RASA objects alongside IRR AS-SET
       queries.

   2.  Validate RASA signatures using standard RPKI validation libraries
       (e.g., rpki-client, Routinator).

   3.  Apply RASA flags during AS-SET expansion.  Honor doNotInherit to
       prevent transitive inclusion.  Prefer authoritative RASA member
       lists over IRR data.  Fall back to IRR data when RASA is
       unavailable.

   4.  Log validation failures and policy decisions for auditing.

13.3.  Monitoring and Alerting

   Operators SHOULD monitor RASA validation status:

   *  Track RASA objects for critical AS-SETs (content providers, Tier-1
      providers) and alert on validation failures.

   *  Monitor RASA object expiration dates and alert before expiry.

   *  Log fallback events when reverting to IRR data due to missing or
      invalid RASA objects.

   *  Track unexpected AS-SET membership changes detected during RASA
      validation.

14.  Relationship to ASGroup

   RASA builds upon the foundational work of Snijders and Korsb√§ck, who
   proposed the ASGroup mechanism [I-D.asgroup] in 2022.  While ASGroup
   introduced important concepts for RPKI-based AS-SET authorization, it
   expired without achieving deployment.  RASA represents an evolution
   of these ideas with practical deployment considerations.

14.1.  ASGroup Innovations

   ASGroup introduced several important concepts that influenced RASA:

   *  RPKI-signed AS group objects providing cryptographic verification
      of membership.

   *  Separation between the containing AS (group owner) and member
      ASes.

   *  The concept of controlling transitive inclusion through the
      referenceable flag.

   *  Recognition that IRR-based AS-SETs lack cryptographic
      verification.






Abrahamsson              Expires 28 August 2026                [Page 28]

Internet-Draft          RPKI AS-SET Authorization          February 2026


14.2.  RASA Simplifications

   RASA addresses several factors that contributed to ASGroup's
   complexity:

   *  *Removed opt-out mechanism:* ASGroup's separate Opt-Out Listing
      objects added significant complexity to validation and reasoning.
      RASA eliminates opt-out in favor of explicit membership
      authorization via the member list.

   *  *Simplified flag semantics:* ASGroup's referenceable boolean was
      replaced with doNotInherit, providing more intuitive control over
      transitive inclusion.

   *  *IRR compatibility:* RASA is designed to work alongside existing
      IRR infrastructure during transition, whereas ASGroup appeared to
      be a complete replacement.

   *  *Clear deployment path:* RASA includes phased deployment guidance
      with incremental adoption stages.

15.  IRR Operator Considerations

   The introduction of RASA objects raises several open questions
   regarding the relationship between RPKI-authenticated AS-SET data and
   existing IRR database entries.  This section documents known issues
   and potential approaches without making definitive recommendations,
   as operational experience is required to determine best practices.

15.1.  IRR Object Deletion vs Retention

   When an AS-SET owner publishes a RASA object with the authoritative
   flag set, questions arise regarding the fate of the corresponding IRR
   AS-SET:

   *  *Deletion:* Should the IRR AS-SET be deleted?  Issues include:
      lack of cryptographic deletion in IRR, persistence in mirrors, and
      potential breakage of legacy tools that do not yet support RPKI.

   *  *Retention:* Should the IRR AS-SET be kept indefinitely?  Issues
      include: potential confusion when IRR and RASA data diverge, and
      validators needing clear precedence rules.

   *  *Deprecation markers:* Can IRR objects be marked as deprecated?
      Issues include: lack of standardized deprecation attributes in
      IRR, and varying support across IRR database implementations.

   Further investigation is needed to understand operational impacts and
   develop migration guidance.

15.2.  IRR-RASA Synchronization

   If both IRR and RASA objects exist for the same AS-SET,
   synchronization questions arise:

   *  *Source of truth:* When RASA authoritative flag is set, validators
      use RASA.  However, IRR databases may serve stale data to legacy
      clients.  Is dual maintenance required during transition?

   *  *Automated updates:* Should IRR databases automatically generate
      AS-SET objects from RASA data?  This would require significant IRR
      infrastructure changes and trust in RPKI validation.

   *  *Conflict detection:* Should validators detect when IRR and RASA
      data differ significantly?  What actions should be taken on
      detection?

   Operational deployment experience is needed to inform synchronization
   strategies.

15.3.  Transition Period Management

   The expected 5-10 year transition period raises operational
   questions:

   *  *Gradual adoption:* How should operators manage AS-SETs where some
      members have RASA objects and others do not?

   *  *Tool compatibility:* How should networks handle tools that only
      query IRR during the transition?

   *  *Documentation:* What documentation should accompany RASA objects
      to guide IRR operators and users?

   These questions require community feedback and operational experience
   before standardization.

16.  Failure Modes

   This section documents expected failure modes and recommended
   responses during RASA deployment and operation.

16.1.  Signature Validation Failures

   When RASA signature validation fails, operators SHOULD:

   *  Fall back to IRR data for the affected AS-SET if configured for
      backward compatibility.

   *  Log the validation failure with specific error details (expired
      certificate, invalid signature, revocation, etc.).

   *  Alert operations staff if the failure affects critical AS-SETs and
      strict RASA mode is enabled.

   *  Continue processing other AS-SETs; do not fail the entire route
      filter generation.

16.2.  Repository Availability Failures

   When RPKI repositories are unavailable:

   *  Use cached RASA data if available and within validity period.

   *  Fall back to IRR data with appropriate logging if cached data is
      unavailable or expired.

   *  Retry repository queries with exponential backoff.

   *  Alert on persistent repository failures affecting validation.

16.3.  Policy Violation Failures

   When RASA validation detects policy violations (unauthorized AS-SET
   inclusion):

   *  Exclude the violating ASN from the route filter expansion.

   *  Log the violation with details of the unauthorized inclusion.

   *  Alert network operations if the violation involves high-value AS-
      SETs or appears to be an attack.

   *  Provide configuration option to warn-only vs. reject for gradual
      policy adoption.

16.4.  Chain Protection Failures

   When an AS-SET in the expansion chain lacks protection:

   *  Process the chain normally if the unprotected AS-SET has
      doNotInherit set (intentionally unprotected).

   *  Log a warning when encountering unprotected AS-SETs in otherwise
      protected chains.

   *  Provide option to require full chain protection for high-security
      environments.

   *  Document the specific unprotected AS-SETs for follow-up with
      operators.
